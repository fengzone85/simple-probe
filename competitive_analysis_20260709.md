# Simple Probe 竞品分析

> 本文档为内部选型参考，记录主流自托管监控工具的设计取向与适用场景。
> 不同项目有不同的取舍，本文不排座次，只尽量客观地呈现差异，供项目定位与文档写作参考。
>
> 成文时间：2026-07-09

---

## 一、项目概览

| 项目 | 技术栈 | 定位 | 部署方式 | 特点 |
|------|--------|------|---------|------|
| **哪吒 Nezha** | Go + Vue | 监控 + 运维（O&M） | 一键脚本 / Docker | 功能最全，社区极大 |
| **Komari** | Go + Node.js | 监控 + 远程控制 | 一键脚本 / Docker / 二进制 | 轻量、美观、有主题 |
| **Pulse** | Go + Astro | 纯监控 | 二进制 / Docker | 实时推送、跨平台、自动更新 |
| **Simple Probe** | Node+SQLite / Python Agent | 纯监控（无控制） | Docker / 原生 systemd | 极简、隐私优先、无指令通道 |

四方都属于"自托管、可私有部署"的监控工具，差异主要在**功能广度**与**安全取向**两个轴上。

---

## 二、逐项对比

### 2.1 功能范围

- **哪吒**：最广。系统状态、SSL 证书、TCP、Ping、网站监控、告警推送、计划任务、**Web 终端**。
- **Komari**：系统 / 网络 / 远程控制 / 主题 / 多语言。
- **Pulse**：系统指标、TCPing 延迟、IPv6、实时 SSE 推送、自定义 CSS/JS。
- **Simple Probe**：系统指标（CPU / 内存 / 磁盘 / 负载 / 网络 / 流量 / 温度 / Swap / 开机时长 / 网络质量自测）。

> 说明：Simple Probe 当前不提供 SSL / 网站 / TCP 类监控，也不提供任何控制能力。这是主动的取舍，见第四节。

### 2.2 安全模型

各项目对"被控端能做什么"的取向不同，这是最值得留意的差异：

| 能力 | 哪吒 | Komari | Pulse | Simple Probe |
|------|------|--------|-------|--------------|
| 命令执行通道 | ✅ Web 终端 + 计划任务 | ✅ 远程控制 | 服务端可 pull（Agent 开端口） | **无** |
| 被控端入站 | 需可达 | 需可达 | 可主动拉取（开端口） | **零入站** |
| 指纹采集 | ✅ GeoIP（IP/位置） | 较深 | ✅ IP/位置/CPU 型号 | **不采集** |
| 认证 | 社区版 2FA | 用户名 + 密码 | Bearer + 共享密钥 | Admin Token + TOTP + 可选只读 |

**关于指令通道**：哪吒的 Web 终端、Komari 的远程控制，都是便利的运维能力，但也意味着被控端需要保留指令通道与执行权限——这是功能与攻击面之间的自然权衡，并非"缺陷"。Simple Probe 选择了另一条路：被控端仅出站上报，不接收任何指令，不开放入站端口。两种取向各有适用场景。

**关于 Komari 的滥用提示**：Komari 官方文档中自带提醒，其远程控制功能存在被未授权滥用的风险，并引用了第三方安全分析。这同样是"带控制能力"这一设计取向的伴随代价，记录于此以供客观判断。

**关于指纹采集**：哪吒使用 GeoIP 数据库、Pulse 采集 IP/位置/CPU 型号；Simple Probe 刻意不采集任何可用于指纹识别的数据（IP、地理位置、MAC、进程列表等），仅上报匿名化的系统负载指标。

### 2.3 被控端资源占用

| 形态 | 常驻内存 | 说明 |
|------|---------|------|
| 哪吒 / Komari / Pulse（Go Agent） | ~5–10 MB | 静态编译二进制，分发轻 |
| Simple Probe（Docker 形态，含 Docker 守护进程） | ~65–150 MB | Docker 守护进程本身占 50–120 MB |
| Simple Probe（原生 systemd 形态） | ~15–25 MB | 裸 Python，无 Docker 开销 |

> 注：Go 原生二进制在分发体积上更轻；Simple Probe 的原生 systemd 形态已接近同量级，且免去 Docker 依赖。对 256–512 MB 的小内存机器，原生形态比 Docker 形态约省 125 MB 可用内存。

### 2.4 部署与生态

- **哪吒**：最成熟。一键脚本、Docker、1Panel 应用商店上架、多前端项目、Weblate 多语言、主题系统、数万 Star。
- **Komari**：一键脚本、Docker、二进制、1Panel 上架、云应用市场、Telegram 群组。
- **Pulse**：单二进制、Docker、自动更新、一键备份/迁移/回滚脚本。
- **Simple Probe**：Docker、原生 systemd（install.sh / uninstall.sh）；项目处于早期，生态尚在积累。

## 三、适用场景建议

以下仅为中性建议，按需求匹配工具：

- **需要远程操作机器、要功能全面**（SSL/网站/TCP 监控、Web 终端）：哪吒更合适。
- **偏好轻量美观、且需要远程控制能力**：Komari 更合适。
- **要实时丰富指标、跨平台、零服务器部署**：Pulse 更合适。
- **只需查看机器状态、在意资源占用 / 隐私 / 不想被控**：Simple Probe 更合适。

## 四、Simple Probe 的主动取舍

为保持极简与安全取向，Simple Probe 主动放弃了一些常见功能：

**主动不做的：**
- 控制功能（Web 终端 / 计划任务 / 远程 shell）
- SSL 证书 / 网站 / TCP 端口监控
- 炫酷主题与高度可定制 UI

**主动选择的：**
- 无指令通道：被控端纯出站，服务端无法连回，无执行接口
- 极简可审计：服务端核心约 800 行，Agent 约 200 行纯 stdlib，半天可审完
- 数据最小化：不采集任何指纹数据
- 双形态部署：Docker（隔离优先）与原生 systemd（资源优先）共用同一套 Agent 代码

这些取舍让 Simple Probe 在"安全姿态"与"资源占用"上形成自己的位置，但也意味着它在功能广度上不追求与哪吒对齐。

## 五、用户群体定位

### 核心受众

1. **安全偏执型运维**：在意被控端是否保留执行能力，希望即便凭证泄露也不会导致机器被控。
2. **小内存机器持有者**：256–512 MB 的小鸡，跑不起 Docker 守护进程；原生 systemd 形态约 15–25 MB。
3. **隐私优先者**：不希望监控工具采集 IP、地理位置、硬件型号等指纹。
4. **审计 / 极简需求者**：希望自行审阅 Agent 行为，代码量小、可读。

### 非目标受众

- 需要 Web 终端 / 远程 SSH 的用户（建议哪吒 / Komari）
- 需要 SSL / 网站 / TCP 监控的用户（建议哪吒）
- 追求最炫 UI 与主题定制者（建议 Komari / 哪吒）
- 要求最省心、生态最成熟的一键部署者（建议哪吒，已上架 1Panel）
